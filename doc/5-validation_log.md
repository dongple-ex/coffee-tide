## 1. 예외 상황 발생 시나리오 및 서버/클라이언트 대응 로직

서버 API Route 및 어댑터 레벨에서 발생하는 다양한 오류를 안전하게 격리하고 사용자에게 올바른 안내를 전달하는 흐름을 사전에 규정합니다.

### 1.1 인증 및 세션 만료 예외 (Session Expired / Unauthorized)
- **발생 위치**: `/api/mails`
- **서버 처리**:
  - 세션 쿠키가 누락되었거나 쿠키 복호화 실패 시 즉시 `HTTP 401 Unauthorized` 상태 코드와 에러 JSON 응답을 반환합니다.
- **클라이언트 처리**:
  - API 호출 결과 상태 코드가 `401`인 경우, 클라이언트의 전역 로그인 세션 상태를 로그아웃 처리하고, 강제로 로그인 화면으로 화면을 전송하여 안전한 재로그인을 유도합니다.

### 1.2 Microsoft Graph API Rate Limit 초과 (HTTP 429 Too Many Requests)
- **발생 위치**: `OutlookAdapter.fetchRecent()`
- **서버 처리**:
  - Graph API 호출 도중 `429` 오류를 캐치하는 경우, 응답의 `Retry-After` 헤더 값을 파싱하여 수용 대기 시간을 산정합니다.
  - 로그 시스템에 `[Warning] Microsoft Graph API Rate Limit Exceeded. Retry-After: {seconds}` 로그를 적재하고, 사용자에게는 즉시 요청 실패 에러 대신 안전하게 빈 리스트 `[]`와 함께 헤더 지연 상태값을 전달합니다.
- **클라이언트 처리**:
  - 화면 상단에 경고 배너로 "현재 Microsoft 서버와의 통신이 원활하지 않아 잠시 대기 중입니다. 5초 후 자동으로 갱신됩니다." 형태의 유도 모달/텍스트를 배치합니다.

### 1.3 데이터 가공 처리 에러 (Parsing Error)
- **발생 위치**: `mapGraphMessageToUnified()`
- **서버 처리**:
  - 특정 메일 객체의 데이터 일부(예: `body` 객체가 null이거나 발신자 주소 객체 누락)가 훼손된 경우, `try-catch` 블록으로 예외를 국소화합니다.
  - 예외 발생 항목만 스킵하거나 기본값(Fallback)으로 대체하여 정상 데이터가 포함된 전체 목록의 반환에 영향을 주지 않도록 격리합니다.

## 2. 세부 검증 시나리오 및 통합 테스트 절차

개발이 완료된 시점에 순차적으로 동작을 확인하는 시나리오 목록입니다.

### Scenario A: Mock Mode 단독 실행 및 UI 인터랙션 검증
- **검증 환경**: `.env.local` 에 `MOCK_MODE=true` 로 설정.
- **순서**:
  1. `npm run dev` 명령으로 로컬 개발 서버 기동.
  2. 브라우저로 `http://localhost:3000`에 접속.
  3. 로그인 전 Welcome 화면에서 "체험 모드로 시작하기" 혹은 "로그인" 버튼 클릭 시, MS 로그인 인증창을 타지 않고 즉시 대시보드 화면이 로드되는지 확인.
  4. 대시보드 카드(Bento Grid) 내에 `MOCK_MAILS`의 메일 3건이 정상 노출되는지 확인.
  5. 메일 카드를 클릭했을 때 해당 메일의 원본 웹 주소(`url` 필드값)로 새 창 이동이 정상 작동하는지 검증.
  6. "오늘 뭐 해야 해?" 와 같은 프롬프트 박스 입력 시, 에러 없이 AI 비서 컴포넌트 내부에 응답 대기 상태 스켈레톤(Skeleton UI)이 연출되는지 체크.

### Scenario B: Microsoft OAuth 실환경 인증 통합 검증
- **검증 환경**: `.env.local` 에 `MOCK_MODE=false` 및 Azure App Key 입력 완료 상태.
- **순서**:
  1. 메인 화면 로그인 버튼 클릭 후, Microsoft 로그인 포털(`https://login.microsoftonline.com/...`)로 리다이렉트가 매끄럽게 처리되는지 검증.
  2. 개인 또는 연동할 회사 계정으로 자격 증명 완료 후, `/api/auth/callback` 주소로 리다이렉트되어 최종 토큰을 획득하는지 브라우저 개발자 도구의 Network 탭에서 확인.
  3. 세션 완료 후 홈 화면으로 재진입 시, 쿠키 스토리지 내에 세션 쿠키가 등록되어 보안 유지(HttpOnly, Secure 속성 활성화)를 수행하는지 검증.
  4. 실제 받은 편지함에 임의의 메일을 전송한 후, TimePilot 대시보드 상에서 새로고침 시 방금 수신한 메일 제목과 내용이 올바르게 가져와지고 리스트 최상단에 렌더링되는지 확인.
