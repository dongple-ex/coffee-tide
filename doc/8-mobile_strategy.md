# Part 8: 모바일 전략

> **결정: coffeeTide는 웹 코어를 유지하되, 하이브리드(WebView) 방식의 네이티브 앱 래퍼를 통해 앱 스토어 출시를 진행합니다.**
> 관련 문서: [`00-current-state.md`](./00-current-state.md)(정본 기획), [`phase6_llm_artifacts_spec.md`](./phase6_llm_artifacts_spec.md) §13(데스크톱 전용 제약), [`7-backlog.md`](./7-backlog.md) E3(반응형 점검).

---

## 1. 기본 방침

1. **웹 코어 우선 (Web-Core First)**: `coffeeTide.dongple.kr`에서 동작하는 반응형 웹 앱을 핵심 엔진으로 사용합니다.
2. **하이브리드 앱 래퍼 (WebView App)**: 껍데기(Shell)만 네이티브(예: React Native WebView, Flutter, Capacitor 등)로 구성하고, 속은 웹 페이지를 띄우는 방식으로 앱 스토어(iOS/Android)에 배포합니다.
3. **모바일은 축소판이 아니라 "입력·확인 중심 뷰"**: 모바일에서는 업무를 빠르게 넣고(manual/paste), 브리핑을 확인하고, 완료 처리하는 흐름을 우선합니다.

## 2. 단계별 로드맵

| 단계 | 내용 | 시점 |
| :--- | :--- | :--- |
| M1 | **반응형 웹** — ≤480px 뷰포트에서 반응형 UI 고도화 및 네이티브 앱(WebView) 통신 인터페이스 준비 | 웹 MVP와 동시 |
| M2 | **하이브리드 앱 래퍼 구현** — WebView를 띄우는 네이티브 껍데기(Capacitor, React Native WebView 등) 구성 및 위치/앱 푸시 권한 연동 | 웹 MVP 안정화 후 |
| M3 | **앱 스토어 출시** — App Store(iOS) 및 Play Store(Android) 심사 및 배포 | M2 완료 후 |

## 3. 데스크톱 전용 기능의 모바일 처리

로컬 파일시스템(`fs`) 접근이 필요한 연동은 **서버 = 사용자 PC** 전제이므로 모바일/클라우드 배포에서는 동작하지 않습니다.

| 기능 | 모바일 동작 | 처리 방침 |
| :--- | :--- | :--- |
| `manual` / `paste` 입력 | ✅ 가능 | 모바일의 **1순위 입력 경로**. 입력 폼을 모바일 최상단에 배치 |
| Copilot 브리핑, 완료/보류/삭제 | ✅ 가능 | 서버 경유라 제약 없음 |
| `outlook` / `notion` / `gmail` 연동 | ✅ 가능 | OAuth·API가 서버 경유라 제약 없음 |
| `local_doc` (로컬 문서 폴더) | ❌ 불가 | 연동 카드 숨김 + "데스크톱 전용" 안내 |
| `obsidian` (볼트 스캔·캡처·미러링) | ❌ 불가 | 연동 카드 숨김 + "데스크톱 전용" 안내 |
| `llm` (LLM 산출물 폴더 스캔) | ❌ 불가 | 연동 카드 숨김 + "데스크톱 전용" 안내 ([`phase6`](./phase6_llm_artifacts_spec.md) §13) |
| 네이티브 폴더 선택기 (`/api/util/select-folder`) | ❌ 불가 | 해당 기능 진입점 자체를 미노출 |

판별 방식: User-Agent 스니핑보다 **배포 형태 기준**(§4)으로 기능을 게이팅하고, 화면 크기는 레이아웃에만 사용합니다.

## 4. 배포 형태와의 관계 (중요한 갈림길)

현재 설계는 Next.js 단일 앱이 로컬 파일까지 읽는 구조라, 배포 형태에 따라 기능 집합이 달라집니다.

- **셀프호스팅 (사용자 PC에서 실행)**: 전 기능 동작. 로컬 파일 연동(local_doc/obsidian/llm) 포함. 같은 네트워크의 모바일 브라우저에서 접속하면 모바일에서도 로컬 연동 *결과*는 볼 수 있음.
- **클라우드 배포 (`coffeeTide.dongple.kr`)**: 로컬 파일 연동 3종은 **비활성**. manual/paste + 외부 API 연동(outlook/notion/gmail)만 제공.

→ 정본의 "무연동 우선(manual/paste 1급 소스)" 원칙(G1)이 지켜지면 클라우드 배포판도 그 자체로 완결된 제품이 됩니다. **모바일 전략의 성립 조건이 곧 G1 구현**입니다.

## 5. 모바일 UX 원칙

1. 첫 화면에서 "빠른 업무 추가" 입력이 엄지 도달 범위 안에 있어야 합니다.
2. 카드 액션(완료/보류/삭제)은 터치 타깃 44px 이상, 이모지 버튼에는 `aria-label` 필수 (백로그 E1).
3. 본문 클램프는 탭으로 펼침 (백로그 E2).
4. 30초 폴링은 모바일 배터리/데이터를 고려해 백그라운드 탭에서 중단(Page Visibility API).

## 6. 성공 기준

- 모바일 브라우저에서 1분 안에 업무 1건 등록 → Copilot 브리핑 확인까지 가능하다 (정본 §6과 동일 기준을 모바일에서 충족).
- 데스크톱 전용 연동이 모바일에서 에러가 아니라 명확한 안내로 표시된다.
- ≤480px에서 가로 스크롤 없이 모든 핵심 흐름을 완료할 수 있다.

## 7. 비범위 (Non-Goals)

- 화면마다 네이티브 UI 뷰를 새로 짜는 완전 네이티브 앱 개발 (웹 뷰 방식으로 대체).
- 모바일 전용 별도 프론트엔드 코드베이스 유지 (웹 코어 하나로 통합).
- 모바일 앱 내부에서의 로컬 파일 접근 (데스크톱 전용 기능 유지).
