## 1. 개요 & 배경

현재 coffeeTide는 외부 연동 데이터와 로컬 메모를 바탕으로 할 일을 분류해주는 기능(Copilot 브리핑)을 갖추고 있습니다. 이를 **Lindy.ai** 스타일로 한 단계 발전시켜, 단순 리스트업을 넘어 비서처럼 **시간대별 업무 흐름을 주도적으로 제안**하도록 만듭니다.

### 1.1 MVP 범위에서 제외된 것: AI 실행 에이전트 (Taskade 스타일)

초기 가안에는 `[AI에게 맡기기]` 버튼으로 coffeeTide가 직접 초안 문서를 생성·저장하는 **실행형 에이전트(목표 2)** 가 포함되어 있었습니다. **이 목표는 MVP 범위에서 제외하고, 데스크톱 전용 옵션 기능으로 §7에 후속 과제로 남깁니다.**

**사유: 이미 로컬 도구가 담당하고 있고, 그 결과가 coffeeTide로 되돌아오는 구조가 존재합니다.**

```
[Claude Code / Gemini 등 로컬 LLM 도구]  ← 초안 작성은 여기서
              ↓ 산출물 파일(.md)
[phase6: llm 소스 폴더 스캔]             ← coffeeTide는 "수집"을 담당
              ↓
[UnifiedData 통합 피드 · Obsidian 미러링]
```

coffeeTide가 초안 생성을 직접 떠안으면 이 루프와 기능이 중복되면서, 아래 비용만 새로 발생합니다.

* **제품 원칙 4(부분 실패 허용) 위반** — 초안 생성은 정규식 폴백이 불가능해, 이 앱 최초의 "AI 없으면 못 쓰는 기능"이 됩니다.
* **개인정보 원칙 위반** — 메일 본문에서 파생된 문서 전문을 외부 저장소(Upstash)에 평문 보관하게 됩니다. `src/lib/push/store.ts:19`가 명시적으로 금지하는 패턴입니다.
* **최대 토큰 비용** — 앱에서 가장 무거운 LLM 호출이 됩니다. 백로그 C1이 캐시·쿨다운·킬스위치까지 도입하며 관리해 온 비용 기조와 어긋납니다.

**따라서 MVP에서 coffeeTide의 역할은 "AI가 대신 실행"이 아니라 "무엇을 넘길지 알려주기"로 정의합니다.** 그 접점이 §2.4의 위임 가능 태그입니다.

> 단, 위 세 가지 비용 중 **개인정보 항목은 저장 위치를 Obsidian 볼트로 바꾸면 해소됩니다.** 이 경로를 §7에 후속 과제로 정리했습니다.

## 2. 세부 구현 계획

### 2.1 상황 맞춤형 웰컴 그리팅 (시간 & 날씨 기반)

앱 접속 시 시간과 날씨에 맞춘 환영 메시지를 Copilot 브리핑 첫머리에 제공합니다.

> "비 오는 화요일 아침이네요. 오늘 오전에는 이런 업무에 집중해보면 어떨까요?"

* **위치 획득은 `@capacitor/geolocation`으로 단일화합니다.** 이 플러그인은 웹에서 브라우저 Geolocation API로 자동 폴백하므로, 웹과 앱에 코드를 두 벌 두지 않아도 됩니다. (`hybrid_app_release_guide.md` §2 Step 4-1과 동일한 결정)
  > **구현 노트 (2026-07-22)**: 현재 구현은 `navigator.geolocation` 직접 호출입니다(`WelcomeCard.tsx` — Capacitor 패키지 미설치). 웹 단독 배포에서는 동작이 동일하며, **하이브리드 앱 착수 시점에 `@capacitor/geolocation`으로 교체**해야 합니다(교체 지점은 `WelcomeCard`의 `useEffect` 한 곳).
* **저정밀(coarse)로 충분합니다.** 날씨 조회에 동 단위 이상의 정밀도는 불필요하고, 스토어 개인정보 신고 항목도 가벼워집니다.
* **3단계 폴백 필수** — 원칙 4에 따라 각 단계 실패가 사용자를 막지 않아야 합니다.

  | 상황 | 결과 |
  | :--- | :--- |
  | 위치 허용 + 날씨 조회 성공 | 날씨 + 시간대 그리팅 |
  | 위치 거부 / 조회 실패 | **시간대만으로** 그리팅 |
  | 그리팅 생성 자체 실패 | 그리팅 영역 미표시, 기존 브리핑은 정상 노출 |

* **원칙 5 준수** — 요일·시간대는 추정하지 않고 클라이언트 타임존을 컨텍스트로 전달합니다. "화요일 아침"이 틀리면 비서로서의 신뢰가 한 번에 무너집니다.

### 2.2 날씨 조회 경로 (`GET /api/weather`) — 신규

* 날씨 API 키가 클라이언트에 노출되면 안 되므로 **서버 라우트를 경유**합니다.
* 좌표를 소수점 2자리로 절삭(약 1km)해 캐시 키로 쓰고, 응답을 10~30분 캐시합니다. 무료 티어 호출량 보호 목적이자, **원좌표를 그대로 외부에 보내지 않는 최소화 조치**이기도 합니다.
* 좌표는 **저장하지 않습니다.** 요청 처리에만 쓰고 버립니다.
* 공급자는 **OpenWeatherMap**으로 확정 (§5.1 결정). `WEATHER_API_KEY` 미설정 시 `success:false`를 반환하고 그리팅은 시간대 폴백으로 동작합니다.

### 2.3 Copilot 시스템 프롬프트 개선 (`src/lib/ai/gemini.ts`)

우선순위 나열 방식을 탈피해 **"오전 집중 업무", "오후 소통 업무", "위임 가능 업무"** 등 시간대별·성격별 행동 제안을 하도록 고도화합니다.

프롬프트가 길어지는 만큼 입력 토큰이 증가하므로, 기존 해시 캐시(백로그 C1)가 새 프롬프트에도 그대로 적용되는지 확인이 필요합니다. **프롬프트 버전을 캐시 키에 포함**해야 프롬프트 변경 시 낡은 응답이 재사용되지 않습니다.

### 2.4 "위임 가능(Delegatable)" 태스크 판별

AI 분류 시 "사용자가 직접 해야 하는 일"과 **"로컬 LLM 도구에 넘길 만한 일"** 을 구분해 태그를 부여하고 UI에 표시합니다.

> **정의 변경 주의**: 초기 가안에서 이 태그는 "coffeeTide 내장 AI가 실행할 일"을 뜻했습니다. 목표2 제거에 따라 **"사용자가 Claude Code 등으로 가져갈 후보"** 를 가리키는 의미로 바뀝니다. 실행 버튼이 아니라 **표식(hint)** 입니다.

이 태그는 phase6 루프의 진입점 역할을 합니다. 사용자는 표시된 업무를 로컬 도구로 처리하고, 그 산출물이 `llm` 소스로 다시 수집됩니다.

**데이터 모델 확장 (`src/lib/types/unified.ts`)**

```ts
export interface UnifiedData {
  // ...기존 필드
  delegatable?: boolean;   // 로컬 LLM 도구로 넘길 만한 업무
}
```

* 선택 필드이므로 기존 데이터와 호환됩니다.
* **폴백 경로 대응 필요** — `fallbackEngine.ts`의 `classifyOne()`은 이 필드를 채우지 않습니다. AI 없이 동작할 때 태그가 없는 것은 정상이며, UI는 `undefined`를 "위임 불가"가 아니라 "판별 안 됨"으로 다뤄야 합니다.
* `00-current-state.md` §3 정본 데이터 허브 표에도 반영합니다.

## 3. 제품 원칙 정합성

`00-current-state.md` §2 기준 자체 점검입니다.

| 원칙 | 적용 |
| :--- | :--- |
| 1. 무연동 사용 가능 | ✅ 그리팅·시간대 제안은 수동 입력 데이터만으로 동작 |
| 2. 수동/로컬 데이터 1급 | ✅ `delegatable` 판별은 소스와 무관 |
| 3. 연동은 증강 기능 | ✅ 위치 권한은 선택, 거부 시 §2.1 폴백 |
| 4. 부분 실패 허용 | ✅ 위치/날씨/AI 각 단계 독립 폴백 (목표2 제거로 위반 해소) |
| 5. 날짜·출처 근거 | ✅ 타임존 컨텍스트 전달, 요일 추정 금지 |

**남은 개인정보 이슈 1건**: 날씨 조회를 위해 좌표가 제3자 API로 전송됩니다. 규모는 작지만 coffeeTide가 사용자 데이터를 외부로 내보내는 첫 경로입니다.

현재 개인정보 관련 원칙은 `src/lib/push/store.ts:19` 코드 주석에만 존재하고 `00-current-state.md`에는 해당 절이 없습니다. **이 기회에 정본 문서로 승격시킬 것을 권합니다** — 스토어 출시 시 개인정보처리방침 작성에도 그대로 필요한 내용입니다.

## 5. 논의 및 결정 사항 (Open Questions → 구현 시 결정 기록)

**5.1. 날씨 API 공급자** — ✅ **결정: OpenWeatherMap** (2026-07-22)
무료 티어 한도, 한국 지역 정확도, 상업적 이용 약관을 함께 봐야 합니다. 후보: Open-Meteo(키 불필요·상업적 이용 무료), 기상청 단기예보 API(국내 정확도 우수, 격자 좌표 변환 필요), OpenWeatherMap.
→ 구현은 **OpenWeatherMap + `WEATHER_API_KEY`** 채택 (`src/app/api/weather/route.ts`, 한국어 응답 `lang=kr` 지원). 키 미설정 시 시간대 폴백으로 동작하므로 키가 없어도 앱은 정상입니다.

**5.2. 그리팅 문구 생성 방식** — ✅ **결정: (a) 템플릿** (2026-07-22)
날씨·시간 조합에 대해 **(a) 템플릿 문구 테이블**을 쓸지, **(b) Gemini로 매번 생성**할지.
→ **(a) 템플릿으로 구현** (`WelcomeCard.tsx`의 시간대 테마 + 날씨 문구 조합). 비용 0·지연 0·폴백 불필요. 그리팅은 브리핑 본문이 아니라 인사말이라, 비용 대비 효용이 낮습니다.

**5.3. 개인정보 원칙의 정본 승격 범위** — 📋 **미결 (오픈 유지)**
`00-current-state.md`에 개인정보 절을 신설한다면 어디까지 담을지 — 외부 전송 목록, 외부 저장 금지 항목, 보관 기간. §3 참조. 스토어 출시(개인정보처리방침 작성) 전까지 결정 필요.

**5.4. 위치 권한 요청 시점** — ⚠️ **현 구현: 첫 진입 즉시 요청. 출시 전 재검토 필요 (백로그 I5)**
첫 진입 즉시 요청할지, 그리팅 영역의 "날씨 켜기"를 눌렀을 때 요청할지. 즉시 요청은 이탈률을 높이고, 앱 심사에서도 목적 불명확으로 지적받을 수 있습니다.
→ 현재 구현은 `WelcomeCard` 마운트 시 즉시 요청합니다. 웹 MVP에서는 허용하되, **앱 스토어 심사 전에 옵트인(버튼 클릭 시 요청) 방식으로 전환을 재검토**합니다 — `7-backlog.md` I5로 등록.

## 6. 기대 효과 및 성공 기준

* **기대 효과**: 사용자는 할 일 목록을 받는 것을 넘어, **"지금 시간대에 무엇부터"** 라는 판단까지 위임할 수 있습니다. 또한 위임 가능 표식을 통해 로컬 LLM 도구로 넘길 업무를 즉시 골라낼 수 있습니다.
* **성공 기준**:
  - 복잡한 수동 입력 업무를 Copilot이 시간대별로 묶어 브리핑한다.
  - 위치 권한을 **거부해도** 그리팅이 시간 기반으로 정상 표시된다.
  - `GEMINI_API_KEY` 미설정 상태에서도 그리팅과 기존 브리핑이 동작한다. *(원칙 4 회귀 방지)*
  - 그리팅에 표시되는 요일·시간대가 사용자 로컬 타임존과 일치한다.
  - 위임 가능으로 표시된 업무를 로컬 도구로 처리한 산출물이, phase6 `llm` 소스로 다시 수집된다. *(루프 완결 확인)*

## 7. 후속 과제 (v2 후보): AI 초안 생성 — 데스크톱 전용 옵션

> **상태**: 📋 후속 과제. MVP(§2) 완료 후 재검토.
> **전제**: Obsidian 연동이 활성화된 데스크톱 환경에서만 동작합니다. phase6과 동일한 티어의 제약입니다.

### 7.1 왜 되살릴 수 있는가

§1.1에서 목표2를 제외한 근거 3개 중, **저장 위치를 Obsidian 볼트로 바꾸면 개인정보 항목이 해소됩니다.** 초안이 사용자 기기 밖으로 나가지 않기 때문입니다.

> Gemini로의 **전송**은 어느 방식이든 동일합니다. 현재도 분류 시 태스크 본문이 Gemini로 갑니다. 이 방식이 새로 늘리는 외부 노출은 없고, 문제가 됐던 것은 외부 **보관**이었습니다.

또한 볼트 쓰기는 **이미 구현되어 검증된 경로**입니다. `src/lib/adapters/obsidian.ts`가 빠른 캡처 append, 완료 write-back, LLM 다이제스트 노트 생성을 모두 수행하고 있습니다. 신규 저장소·TTL 정책·초안 뷰어 API가 전부 불필요해집니다.

**부수 효과**: 볼트에 쓴 초안 `.md`는 다음 스캔에서 `obsidian` 소스로 다시 잡혀 통합 피드에 자연스럽게 합류합니다. 별도 저장소에 고립되지 않습니다.

### 7.2 남은 제약 (해소되지 않음)

| 제약 | 내용 |
| :--- | :--- |
| **데스크톱 전용** | `fs.writeFile`은 Vercel에서 동작하지 않습니다. 모바일 앱에서는 이 기능이 노출되지 않습니다. |
| **원칙 4 (부분 실패 허용)** | 초안 생성은 정규식 폴백이 불가능합니다. → **연동이 없으면 버튼을 아예 노출하지 않는 것**으로 대응합니다. 없는 기능은 원칙을 위반하지 않습니다. |
| **비용** | 앱에서 가장 무거운 LLM 호출입니다. 백로그 C1의 쿨다운·킬스위치를 동일하게 적용해야 합니다. |

### 7.3 구현 스케치

**저장 위치**
```
<vault>/coffeeTide_초안/YYYY-MM-DD_제목.md
```
`obsidian.ts`의 `LLM_DIGEST_DIR`(`coffeeTide_LLM`) 상수 옆에 `DRAFT_DIR` 상수를 나란히 둡니다.

**어댑터 확장** — `ObsidianAdapter.writeLlmDigest()`와 동일한 형태(static, `vaultPath` 인자, `mkdir` 후 idempotent write)로 메서드 하나를 추가합니다.

```ts
static async writeDraft(
  vaultPath: string,
  dateKey: string,
  title: string,
  markdown: string
): Promise<string>   // 반환: 볼트 상대 경로
```

**데이터 모델** — 신규 필드가 필요 없을 수 있습니다. 반환된 상대 경로로 `fetchRecent()`가 이미 쓰는 방식의 URL을 그대로 구성할 수 있기 때문입니다.
```
obsidian://open?file=<encodeURIComponent(상대경로 확장자 제외)>
```
태스크에 초안을 연결해 두려면 `aiDraftPath?: string`(볼트 상대 경로) 한 개면 충분합니다. **Redis 기반의 `aiDraftId`는 필요 없습니다.**

**API (`POST /api/tasks/execute`)**
* 세션 인증 및 태스크 소유권 확인 (LLM 유료 경로)
* Obsidian 연동 미설정 시 **404/400이 아니라 기능 미노출**이 정상 — UI가 버튼을 감춥니다
* `quotaCooldownUntil` 및 킬스위치 연동 (백로그 C1과 동일)
* 원칙 5에 따라 현재 날짜·타임존을 프롬프트 컨텍스트로 전달
* 실행 시간 대응: **스트리밍 응답 권장** (문서 생성은 수 초~수십 초)

**UI**
* Obsidian 연동 ON + `delegatable` 태스크에만 `[🤖 초안 맡기기]` 노출
* 완료 후 `[📄 초안 열기]`로 전환 → `obsidian://` 딥링크
* 실패·타임아웃 시 버튼 원복하여 재시도 가능

### 7.4 착수 전 판단 기준

이 기능의 실익은 **"Claude Code를 이미 켜둔 사용자에게 앱 안의 버튼이 얼마나 더 나은가"** 에 달려 있습니다.

* **장점**: 태스크에서 한 번에 — 컨텍스트를 복사해 옮길 필요가 없음
* **단점**: 레포 컨텍스트가 없는 Gemini 단발 호출이라, 로컬 도구 대비 품질이 낮음

→ MVP(§2)의 **위임 가능 태그를 먼저 써보고**, 사용자가 태그된 업무를 로컬 도구로 옮기는 마찰이 실제로 크다고 느껴질 때 착수하는 것이 순서상 맞습니다. 마찰이 크지 않다면 이 기능은 만들지 않는 것이 최선입니다.
